Settlr — Honest Teardown

12 April 2026 · No sugar-coating, no vibes, just what's broken

TL;DR — You built a real product on top of a house of cards

The payment mechanics, LeafLink integration, and on-chain program are genuinely impressive for a solo build. The problem: everything runs on devnet with an in-memory database that evaporates on restart. Your landing page claims $67K settled and "full BSA/AML screening" — neither is true. You have 3 pricing tiers that don't exist in code, a KYC system that silently says "verified" to everyone, and an API key hash function a CS101 student would fail for. The bones are real. The skin is lies.

🔥 Critical — Would break the product for a real user

#1 Entire app is hardwired to Solana Devnet Critical
Every payment path hardcodes https://api.devnet.solana.com as a string literal. The USDC mint address 4zMMC9srt5Ri5X14GAgXhaHii3GnPAEERYPJgZJDncDU is the devnet USDC mint, hardcoded in 15+ files. Real USDC on mainnet is a completely different address. No real dollar has ever moved through this system, and it can't without changing dozens of hardcoded values.
InvoicePayClient.tsx:37 · CheckoutClient.tsx:82,104 · 12+ API routes
Fix: Create a single src/lib/constants.ts with USDC_MINT, RPC_URL, and IS_MAINNET driven by env vars. Find-and-replace every hardcoded occurrence. This is a 2-hour fix that's blocking your entire mainnet story.
#2 Database silently falls back to in-memory Maps Critical
When Supabase isn't configured, supabase.ts creates a client pointing at https://placeholder.supabase.co. Then db.ts falls back to Map<> objects for all storage — merchants, payments, invoices, sessions. Every server restart or Vercel cold start loses all data. Checkout sessions are also stored in a plain Map. This isn't a dev convenience — it's data loss waiting to happen.
supabase.ts:24-26 · db.ts:293 · checkout/store.ts:8
Fix: Remove the in-memory fallback entirely. If Supabase isn't configured, fail loudly at startup. Run the purchase_orders migration. Make the app refuse to boot without a real database.
#3 KYC results are received and thrown away Critical
The Sumsub webhook handler receives KYC verification results — handleApplicantReviewed, handleApplicantActionPending, handleApplicantActivated — but all Supabase persistence logic is commented out with // TODO: markers. Results are logged to console then discarded. A user who completes KYC would appear unverified forever.
kyc/webhook/route.ts:110-179
Fix: Uncomment the Supabase inserts. Add a kyc_verifications table. Wire the status check to read from it instead of calling Sumsub every time.
#4 KYC silently disabled when env var missing Critical
If SUMSUB_APP_TOKEN isn't set, the KYC status route returns { verified: true } for every user. Combined with #3, your landing page claims "full BSA/AML screening on all users" when in reality: no env var = everyone's verified, env var set = results are thrown away anyway.
kyc/status/route.ts:40-44
Fix: Return { verified: false, error: "KYC not configured" } when the token is missing. Never default to verified.

😬 Embarrassing — Would tank a demo or investor meeting

#5 "$67K settled" is a hardcoded animation Embarrassing
The SocialProof component shows a counter animating up to "$67K+ Settled this quarter". It's a hardcoded number in a config array, not queried from any data source. Testimonials section repeats the claim. Any investor who asks "show me the on-chain proof" would find devnet transactions worth $0. This is the kind of thing that ends meetings.
SocialProof.tsx:41 · Testimonials.tsx:66
Fix: Either pull real settlement volume from on-chain data (even if it's $0), or label it as "demo" / "testnet". Or just remove it until you have real numbers.
#6 Three pricing tiers that don't exist Embarrassing
Explorer (free), Settlement ($49/mo), Enterprise ($199/mo) — complete with feature lists like "Full compliance suite", "Volume discounts", "Multi-entity support", and "Dedicated manager." There is zero tier enforcement in the codebase. No billing integration. No upgrade flow. No plan check on any endpoint. Every user gets the same thing regardless of "plan."
Pricing.tsx:8-46
Fix: Either implement basic tier gating (even a simple Stripe subscription check), or simplify to a single "1% per transaction" pricing model that matches reality. The latter aligns with your actual business model.
#7 Demo API key gives anyone full merchant access Embarrassing
Hardcoded DEMO_API_KEY = "sk_test_demo_xxxxxxxxxxxx" maps to a DEMO_MERCHANT_STUB with a real devnet wallet. Anyone who guesses or discovers this key gets full merchant access — create invoices, view settlements, the works.
db.ts:1215-1221
Fix: Gate demo mode behind a DEMO_MODE=true env var that's only set in development. In production, there should be no backdoor keys.
#8 OTC desk returns fake wire instructions Embarrassing
The OTC quote endpoint returns bankName: "Partner Bank (via OTC Provider)" and routingNumber: "Contact support for details". This is a fully scaffolded endpoint with no actual OTC provider backing it. If a cannabis operator tries to do a $50K wire, they'd hit a dead end.
otc-quote/route.ts:86-92
Fix: Remove the route until you have a real OTC partner. Or gate it behind an env var / feature flag. Don't serve fake banking details.
#9 Auto-offramp is entirely simulated Embarrassing
In-memory arrays. No Sphere, no Bridge, no real offramp provider. All state lost on restart. The dashboard settings page lets merchants "configure" auto-offramp that goes nowhere.
auto-offramp/route.ts:47-61
Fix: Label it as "Coming soon" in the UI. Don't let merchants configure settings for a feature that doesn't work.
#10 Explorer links all say "Solana Devnet" Embarrassing
Every Solana Explorer link in the settlements page appends ?cluster=devnet. The UI literally renders "Solana Devnet" as a label. A customer clicking "View on Explorer" would see it's a devnet transaction worth nothing.
settlements/page.tsx:82,255,261 · db.ts:809
Fix: Drive cluster param from the same IS_MAINNET constant from Fix #1.
#11 Analytics page is a placeholder Embarrassing
"Analytics Coming Soon" with a link to the dashboard. It's in the sidebar nav. If an investor clicks it during a demo, you're done.
dashboard/analytics/page.tsx:13
Fix: Either build a basic chart (settlement volume over time from the API you already have), or remove it from the nav.
#12 "Powered by" logos imply partnerships that don't exist Embarrassing
LogoBar shows Solana, USDC, and Squads Protocol logos as a marquee "powered by" section. These are technologies you use, not partners or customers. Positioned like a trust bar, it implies endorsement. None of these organizations have endorsed Settlr.
LogoBar.tsx:8-11
Fix: Change label from "Powered by" to "Built on" — accurate and still impressive.

⚠️ Weak — Functional but clearly unfinished

#13 3 different RPC env var names Weak
NEXT_PUBLIC_SOLANA_RPC, NEXT_PUBLIC_SOLANA_RPC_URL, and SOLANA_RPC_URL are used interchangeably across different files. A deployer who sets one won't cover the others. Payments will silently fall back to devnet on routes that check a different env var name.
Fix: Pick one name. Alias the others to it in a single config module. 15-minute fix.
#14 API key hashing is trivially weak Weak
hashApiKey() uses hash = ((hash << 5) - hash) + char — a string hash from the 1990s. Not cryptographic. High collision probability. If you had real merchants with real API keys, this would be a security vulnerability. Production should use SHA-256 or bcrypt.
db.ts:1207-1213
Fix: Replace with crypto.createHash('sha256').update(key).digest('hex'). Three lines.
#15 FAQ claims "GENIUS Act compliant" Weak
The FAQ states "Settlr is GENIUS Act compliant and performs full BSA/AML screening on all users." The GENIUS Act regulates stablecoin issuers, not payment platforms. Settlr doesn't issue stablecoins — it routes USDC. This claim is both inaccurate and could draw regulatory scrutiny.
FAQ.tsx:18
Fix: Change to something like: "Settlr uses USDC (a fully-reserved, audited stablecoin) and performs KYB/KYC verification on all merchants." Accurate, still impressive.
#16 LeafLink integration is real — but invisible Weak
The LeafLink client, webhook handler, sync-back mechanism, and callback system are genuinely well-built. But there's no UI to configure or manage the integration. A merchant can't set their LeafLink API key through the dashboard. It's a backend feature with no frontend.
lib/leaflink/client.ts · api/integrations/leaflink/*
Fix: Add a LeafLink settings panel to the dashboard integrations page. Even a simple form for API key + webhook URL display.
#17 Onboarding skips invite token in development Weak
If NODE_ENV isn't explicitly production, anyone can onboard without an invite. On Vercel this is fine (it defaults to production), but a misconfigured deploy or preview branch would be open to anyone.
onboarding/page.tsx:103-106
Fix: Check for a specific ALLOW_OPEN_REGISTRATION flag instead of relying on NODE_ENV.
#18 Onboarding state lives in localStorage Weak
After merchant registration, vault PDA and merchant ID are stored in localStorage. The useOnboardingStatus hook falls back to localStorage when the API errors. Clearing browser data = user appears unregistered. Different browser = different merchant state.
onboarding/page.tsx:253-265 · useOnboardingStatus.ts:62-67
Fix: Use server-side session (Privy auth token → DB lookup) as the source of truth. localStorage as a cache is fine, not as the source of truth.
#19 SDK directory is empty Weak
The README and Architecture doc reference @settlr/sdk v0.6.0 published. The packages/ directory is empty. Either the SDK was removed, or it's published from elsewhere and the workspace link is broken.
packages/
Fix: Either restore the SDK source, or remove the "SDK published" claims until you link to the actual npm package.
#20 Privy embedded wallets are the right call — but untested at scale Weak
Cannabis operators will not install Phantom. Privy embedded wallets (email login → invisible Solana wallet) are exactly right. But: have you tested what happens when a buyer with a Privy wallet needs to off-ramp $45K in USDC to a bank? The on-ramp (MoonPay) works, but the off-ramp is simulated (#9). The loop isn't closed.
Fix: Either integrate a real off-ramp (Sphere or Bridge) or document that off-ramp is manual (transfer to Coinbase/exchange). Don't pretend the loop is closed when it isn't.
#21 No rate limiting on any API route Weak
Invoice creation, checkout session creation, KYC token generation, payment link generation — none have rate limiting. A script could generate infinite invoices, exhaust Sumsub API quota, or spam payment links. Vercel's edge functions have some natural protection, but it's not deliberate.
Fix: Add Vercel KV-based rate limiting (or Upstash) to the critical routes. The @upstash/ratelimit package takes 10 minutes to integrate.

The Honest Verdict

What's real: The Anchor program works. The LeafLink integration is genuinely well-architected. The checkout flow handles embedded wallets and gasless transactions correctly. The MagicBlock PER privacy layer is novel. The Solana Actions/Blinks integration is clever. For a solo build, the scope is impressive.

What's fake: $67K settled. Three pricing tiers. "Full BSA/AML screening." OTC desk. Auto-offramp. Analytics. The SDK. These are claims your marketing makes that your code doesn't back up.

The gap: You built a real payment system and then wrapped it in marketing that oversells what exists. An investor doing 20 minutes of due diligence would find the devnet transactions, the placeholder database, the commented-out KYC. That's worse than having a less impressive product that's honest about what works — because it looks intentionally misleading, not just early-stage.

The fix is not more engineering. It's honesty + the 4 critical fixes above. Change "$67K settled" to "Built and tested on devnet." Change "Full compliance" to "KYC integration in progress." Remove the pricing tiers until they exist. Fix the 4 criticals. Then you have a product that's early but real — which is exactly what an investor or first customer wants to see.

Your product is 6 focused fixes away from being demoable to real cannabis operators. The foundation is solid. Stop lying about the roof.
